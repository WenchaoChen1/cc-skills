# 生成功能设计文档

根据页面截图和需求文档，生成面向 AI 开发的完整功能设计文档。
文档包含 AI 可直接执行的前端 UI 描述、后端接口规格、业务规则与计算公式。

## 使用方式

```
/dev/gen-design-doc <需求文档路径> [截图路径1] [截图路径2] ...
```

例如：
- `/dev/gen-design-doc financial-report`（自动定位 `{features}/financial-report/requirement/`）
- `/dev/gen-design-doc financial-report screenshots/list.png screenshots/detail.png`（带截图）

> 截图路径支持相对路径（相对于当前项目根目录）或绝对路径。
> 若只提供需求文档，则 UI 部分从需求文档中推断。

## 执行步骤

### 第〇步：确定功能目录与输入材料

**若 `$ARGUMENTS` 非空**：
- 第一个参数为需求文档路径（`.md/.txt/.docx`），后续参数为截图路径
- 功能名称优先级：`{features}/xxx/requirement/requirement-doc.md` → 取 `xxx`；其他文件 → 取去掉路径和扩展名的文件名

**若 `$ARGUMENTS` 为空**：
- 扫描 `{features}/` 目录，列出含 `requirement/` 子目录的功能目录
- 若只有一个：提示「检测到 `{features}/{name}/requirement/`，是否基于其中的内容生成设计文档？[Y/n]」
- 若有多个：列出编号清单，提示用户选择
- 若无匹配：提示「未找到需求文档，请先运行 /gen-requirement-doc 生成需求文档」

**重复创建检查**（确定功能名称后执行）：
- 若 `{features}/{name}/dev-design/dev-design-doc.md` 已存在：
  ```
  ⚠️ 文件已存在：{features}/{name}/dev-design/dev-design-doc.md
  建议先运行 /dev/review-design-doc <name> 审查现有设计文档。
  选项：A. 覆盖（原文件被替换）  B. 取消（默认）  C. 查看现有文件内容
  ```
  等待用户选择，默认取消。

### 第一步：读取输入材料

解析 `$ARGUMENTS`：
- **有参数时**：第一个参数为需求文档路径，使用 Read 工具读取（`.docx` 需用 python-docx 提取全文）；后续参数为截图路径，使用 Read 工具逐张读取图片
- **无参数时（自动检测到功能目录）**：读取 `{features}/{name}/requirement/` 目录下所有文件（逐一读取，不遗漏），后续参数为截图路径

**文件读取容错**：
- 需求文档目录或文件不存在 → 提示「缺少需求文档，请先运行 `/gen-requirement-doc <功能名称>`」，不继续后续步骤
- 文件存在但内容为空 → 提示「文件存在但内容为空，请检查对应文件」
- 编码问题（乱码） → 提示「文件编码异常，建议重新保存为 UTF-8」
- 截图路径不存在 → 提示「截图文件未找到：<路径>，将仅基于需求文档生成设计」，继续执行

### 第二步：分析截图，提取 UI 信息

若有截图，对每张截图逐区域分析，提取以下信息：

**布局结构**
- 页面整体分区（头部/侧边栏/主内容区/底部等）
- 各区域的相对位置和尺寸比例
- 是否有弹窗、抽屉、Tab、折叠面板等层叠结构

**组件识别**
- 每个可识别的 UI 组件（表格、表单、图表、按钮、筛选器、卡片等）
- 组件在页面中的位置
- 组件的视觉状态（默认/禁用/选中/加载中）

**数据展示规则**
- 表格的列名、列类型（文本/数字/日期/状态标签/操作按钮等）
- 数字的格式（货币符号、小数位、千分位、正负号颜色）
- 图表类型（折线/柱状/饼图/混合）及坐标轴含义

**交互行为**
- 可点击的元素及点击后的预期效果
- 筛选条件（日期选择、下拉、搜索框等）
- 表格行操作（查看详情、编辑、删除等）
- 分页方式

**色彩语义**
- 关键颜色的含义（如红色=超支、绿色=盈余、蓝色=进行中）
- 图标的语义
- 数据为空时的占位展示

**截图分析完成后，在继续下一步前，必须输出「截图分析存档」**（防止读取需求文档后细节丢失）：

```
【截图分析存档】

▌ 页面分区（截图N）
- 顶部：...
- 主内容区：...
- 底部：...

▌ 表格列（从左到右）
| 列名 | 类型 | 格式 | 对齐 |
|------|------|------|------|
| ...  |      |      |      |

▌ 数字/颜色规则
- 金额：保留X位小数，负数红色
- ...

▌ 交互入口
- 元素A → 点击 → 效果B
- ...

▌ 待确认的细节（截图中模糊或有歧义的部分）
- ?：...
```

> 输出存档后再读取需求文档。生成设计文档时直接引用存档，不重新分析截图。

### 第三步：结合需求文档，推断业务逻辑

对照需求文档内容，补全截图中无法直接看出的信息：

- 每个字段的数据来源（来自哪张表或哪个接口）
- 计算字段的公式（如毛利率 = 毛利 / 收入 × 100%）
- 汇总逻辑（SUM / AVG / 分组规则）
- 筛选条件对应的后端查询逻辑
- 数据的刷新时机（页面加载/手动刷新/定时轮询）

### 第四步：设计后端接口规格

根据页面功能推断所需接口：

对每个接口：
- 确定 HTTP 方法 + 路径（参考项目现有接口风格）
- 设计请求参数（路径参数、Query 参数、请求体字段）
- 设计响应结构，说明每个字段的含义
- 描述后端业务逻辑（查询哪些表、做什么计算、有哪些过滤/排序）
- 列出涉及的数据库表

### 第五步：生成并保存功能设计文档

**保存路径**：`<当前项目根目录>/{features}/{name}/dev-design/dev-design-doc.md`
（若 `{features}/{name}/dev-design/` 目录不存在，先创建）

功能名称的获取优先级：
1. 若第一个参数是 `{features}/xxx/requirement/requirement-doc.md`，取 `xxx` 作为功能名称
2. 若第一个参数是独立的需求文档，取去掉路径和扩展名后的文件名
3. 否则根据功能主题生成英文短名（小写 + 连字符）

## 规则

- **截图与需求分层原则**：UI 细节（布局/样式/格式）以截图为准，业务逻辑（功能点/规则/公式）以需求为准；冲突处在文档中标注差异来源（如「⚠️ 截图与需求差异：截图显示3列，需求描述为4列，此处以截图为准」）
- **不允许留空章节**：若某章节无法从输入材料推断，必须标注 `（需确认）` 并给出合理假设
- **接口必须完整**：每个接口必须包含请求参数、响应结构、后端业务逻辑三部分
- **公式集中到第 6 章**：截图中所有存在计算关系的数值，必须在第 6 章明确公式
- **模板中的示例内容不得照搬**：所有 `<占位符>` 必须替换为实际内容
- 保存后提示：`设计文档已保存至 {features}/{name}/dev-design/dev-design-doc.md → 可运行 /dev/review-design-doc <功能名称> 审查设计，或 /dev/run <功能名称> 直接开发`
