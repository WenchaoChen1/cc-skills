# /dev/common — 公共开发规范

> 本文件定义所有开发 skill（backend-java / backend-python / frontend）的公共规则。
> 每个子 skill 在执行前必须遵守这些规范。也可独立调用 `/dev/common` 查看规范。

---

## 使用方式

```
/dev/common                     # 查看公共开发规范
/dev/common <功能名称>           # 执行准备阶段（读材料 + 探索代码模式），不做实现
```

---

## 规则（所有开发 skill 必须遵守）

### 规则 A：先读后写

- 任何文件动笔之前，**必须先读至少 2 个同类现有文件**作为模式参考
- 禁止凭印象猜测现有代码的包名、类名、字段命名风格
- 先阅读项目的 `CLAUDE.md`，了解技术栈、目录结构、模块组织方式

### 规则 B：改动最小化

- 不修改无关文件；不重构未涉及的代码
- 新增文件优先放进现有模块目录，不新建顶层目录
- 不添加未使用的依赖、不引入不必要的新包

### 规则 C：遇到不确定的设计点，停下来问

- 如果设计文档与现有代码模式有明显冲突，列出冲突点，询问用户如何取舍
- 不要自行决定推翻现有架构
- 如果某个章节设计文档未覆盖且无法从现有代码推断，停下来问

### 规则 D：输出可追溯

- 每个阶段完成后必须输出文件清单（新建/修改）
- 自检结果逐项列出
- 有疑问或取舍的地方标注 `⚠️`

## 准备阶段（通用流程）

> 以下步骤在任何开发 skill 执行前都必须完成。`/dev/run` 作为调度器时会统一执行准备阶段，再分发给子 skill。子 skill 独立调用时需自行执行。

### 第一步：读 CLAUDE.md + 代码规范

读取项目根目录的 `CLAUDE.md`，了解：
- 技术栈（后端语言/框架、前端框架）
- 目录结构与模块组织
- 构建/运行/测试命令

**必须读取代码规范（三层，优先级：子项目 > 项目 > 个人）**：

1. 个人规范（`~/.claude/standards/`）：`common.md` + 对应端规范
2. 项目规范（`.claude/standards/common.md`，如存在）
3. 子项目规范（子项目 `.claude/standards/architecture.md` + `coding.md`，如存在）

> **关键**：后续所有步骤中的具体路径、包名、框架版本、ORM 工具、响应格式等，均以 `CLAUDE.md`、代码规范和实际项目代码为准，不使用任何预设值。

### 第二步：读设计文档

读取 `features/<功能名称>/dev-design/` 目录下**所有文件**，提取并记录：
- [ ] 功能概述（1-2句话）
- [ ] 涉及的后端接口列表（方法 + 路径 + 请求/响应结构）
- [ ] 涉及的数据库表（字段 + 类型 + 约束）
- [ ] 前端页面路由 + 文件结构
- [ ] 业务规则和计算公式（第 6 章）
- [ ] 异常处理规则（第 7 章）

**容错**：
- `dev-design/` 不存在 → 提示「缺少设计文档，请先运行 `/dev/gen-design-doc <功能名称>`」，终止
- 文件存在但为空 → 提示「文件存在但内容为空，请检查」
- 编码异常 → 提示「文件编码异常，建议重新保存为 UTF-8」

### 第三步：读需求文档（补充）

若 `features/<功能名称>/requirement/` 存在，读取所有文件，补充：
- 边界条件与异常规则
- 字段必填/非必填说明
- 权限规则

不存在 → 提示「缺少需求文档，仅基于设计文档开发」，继续执行。

### 第四步：探索现有代码模式

> 根据 CLAUDE.md 中描述的项目结构，找到对应目录读取参考文件。以下仅为**参考维度**，具体路径和文件类型由项目实际结构决定。

**后端（每种涉及的后端语言各读）**：
- 同模块 1 个已有接口层文件（了解路由风格、统一响应格式）
- 同模块 1 个已有业务逻辑层文件（了解事务、异常处理）
- 同模块 1 个已有数据模型文件（了解 ORM 注解/字段风格）
- 同模块 1 个已有 DTO/Schema 文件（了解请求/响应对象定义）

**前端**：
- 1 个结构相似的已有页面文件（了解 hooks/状态管理、组件拆分粒度）
- 1 个已有 API service 文件（了解请求封装、类型定义方式）
- 路由配置文件（了解注册方式和位置）

## 通用自检清单

| 检查项 | 说明 |
|--------|------|
| 包名/路径正确 | import 无遗漏，模块路径与现有代码一致 |
| 命名风格一致 | 参照同模块现有文件的命名惯例 |
| 无编译/语法错误 | 模块路径、类型定义、括号匹配 |
| 无未使用的 import | 清理多余引入 |
| 改动范围受控 | 未修改无关文件 |

## 常见错误预防

| 错误类型 | 预防措施 |
|---------|---------|
| 包名/模块路径猜错 | 开始前读同模块现有文件确认路径 |
| 前后端字段名不一致 | 逐字段核对后端 DTO/Schema 和前端 interface |
| 架构冲突自行决定 | 发现冲突时停下列出，询问用户后再动手 |
| 映射层漏转字段 | 使用项目现有的映射工具和配置风格 |
