# 审查功能设计文档

读取 `features/<name>/` 下的产品需求文档和开发设计文档，检查设计是否完整覆盖需求、接口是否规范、前后端是否一致。

## 使用方式

```
/dev/review-design-doc <功能名称 或 文档路径>
```

例如：
- `/dev/review-design-doc financial-dashboard` — 按功能名称，自动读取 `features/` 下对应目录
- `/dev/review-design-doc 财务看板`
- `/dev/review-design-doc path/to/design-doc.md` — 直接指定设计文档路径
- `/dev/review-design-doc design.md requirement.md` — 同时指定多个文档

> 传入功能名称时，自动读取 `features/<name>/requirement/` 和 `features/<name>/dev-design/` 目录下所有文件
> 传入文件路径时，直接读取指定文件，不依赖 `features/` 目录结构

---

## 执行步骤

### 第一步：读取所有材料

1. 若 `$ARGUMENTS` 非空，先判断参数类型：
   - **情况 A — 指定了文档路径**（参数含文件扩展名如 `.md`、`.docx`、`.txt`，或指向已存在的文件路径）：
     - 直接读取指定文件作为待审查材料，**不扫描 `features/` 固定目录**
     - 支持多个文件路径（空格分隔），依次读取；截图路径一并读取
     - 审查报告保存到文档所在目录的同级 `reviews/` 下
     - **跳过下方第 3 条的目录读取逻辑**
   - **情况 B — 仅指定功能名称**（参数不含文件扩展名，如 `xxx` 或 `features/xxx`）：
     - 解析功能名称，继续执行下方第 3 条
2. 若 `$ARGUMENTS` 为空：
   - 扫描 `features/` 目录，列出含 `dev-design/` 子目录的功能目录
   - 若只有一个：提示「检测到 `features/<name>/dev-design/`，是否审查该功能？[Y/n]」
   - 若有多个：列出编号清单，提示用户选择
   - 若无匹配：提示「未找到设计文档，请先运行 /dev/gen-design-doc 生成设计文档」
3. **仅情况 B 和空参数时执行 — 分阶段读取文件**（避免一次性读取多个大文件导致上下文溢出）：
   - **先读取** `dev-design/` 目录下所有文件，完整提取后输出「设计文档提取清单」（见下方）
   - **再读取** `requirement/` 目录下所有文件（重点关注第 3、5、6 章内容，跳过其他章节）
   - 若有 `req-spec.md`，仅读取与当前审查相关的章节
   - `CLAUDE.md` 用于接口风格判断，重点读取关于后端架构和模块结构的章节（了解分层方式、业务域目录、统一响应格式等）

**文件读取容错**：
- `dev-design/` 目录不存在 → 提示「缺少设计文档，请先运行 `/dev/gen-design-doc <功能名称>`」，不继续后续步骤
- `requirement/` 目录不存在 → 提示「缺少需求文档，跳过需求覆盖检查，仅做设计内部质量审查」，继续执行第三步
- 文件存在但内容为空 → 提示「文件存在但内容为空，请检查对应文件」
- 编码问题（乱码） → 提示「文件编码异常，建议重新保存为 UTF-8」

**读取 `dev-design/` 目录下所有文件后，立即输出「设计文档提取清单」，再继续读需求文档：**

```
【设计文档提取清单】

▌ 接口清单（第 4.1 章）
| 接口名 | 方法 | 路径 |
|--------|------|------|
| ...    |      |      |

▌ 数据表清单（第 5 章）
- 表名：字段列表

▌ 业务规则清单（第 6 章）
- 规则1：...

▌ 异常处理清单（第 7 章）
- 场景：错误码
```

> 提取完毕后才读需求文档，做映射时直接基于上方清单对比，不重新翻阅设计文档全文。

### 第二步：需求覆盖检查（有 `requirement/` 目录时执行）

**读取 `requirement/` 目录下所有文件（重点关注第 3、5、6 章内容）**，提取后输出「需求提取清单」，再做映射：

```
【需求提取清单】

▌ P0/P1 功能点（第 3 章）
- P0: 功能A
- P1: 功能B

▌ 业务规则（第 5 章）
- BR-01: ...

▌ 计算公式（第 6 章）
- 公式1: ...

▌ 边界条件（第 5.4 章）
- 场景1: ...
```

**基于两份提取清单做映射**（不再翻阅原文）：

**功能点覆盖**
- 逐条对比「需求 P0/P1 功能点」与「设计接口清单」，标注 ✅/❌/⚠️

**业务规则覆盖**
- 逐条对比「需求业务规则」与「设计业务规则清单」

**计算公式覆盖**
- 逐条对比「需求计算公式」与「设计文档第 6 章」

**异常场景覆盖**
- 逐条对比「需求边界条件」与「设计异常处理清单」

### 第三步：设计文档内部质量审查

#### 3-A. 接口设计审查

对设计文档第 4 章（后端接口设计）逐个接口检查：

| 检查项 | 判断标准 |
|--------|---------|
| 路径风格 | 与 CLAUDE.md 中描述的项目接口风格一致 |
| HTTP 方法 | GET 用于查询，POST/PUT/PATCH 用于写操作，DELETE 用于删除 |
| 请求参数 | 必填/选填标注明确，类型正确，有说明 |
| 响应结构 | 有完整 JSON 示例，每个字段有类型和说明 |
| 业务逻辑 | 说明了查询哪些表、如何过滤、如何计算 |
| 错误处理 | 在第 7 章有对应的异常场景 |

#### 3-B. 数据模型审查

对设计文档第 5 章（数据模型）检查：

- 每张表是否有主键
- 字段类型是否合理（金额用 DECIMAL，时间用 TIMESTAMPTZ）
- 外键关联是否说明
- 新增字段/表是否有 `created_at`/`updated_at` 审计字段
- 索引是否说明（高频查询字段）

#### 3-C. 前后端一致性检查

对比设计文档第 2 章（UI 设计）和第 4 章（接口设计）：

- 前端表格每一列的 `fieldName` 是否与接口响应字段名完全一致
- 前端筛选条件是否对应接口的 Query 参数
- 前端提交表单字段是否对应接口的请求体字段
- 交互行为（编辑/删除/查看）是否都有对应接口

#### 3-D. UI 完整性审查

对设计文档第 2 章检查：

- 是否有空态设计（查询无结果）
- 是否有加载态设计
- 是否有错误态设计（接口报错）
- 交互行为（2.4 章）是否每条都有描述效果

#### 3-E. 文档质量检查

**去重**：
- 同一接口的业务逻辑在第 4 章接口详情和第 6 章业务规则中重复 → 建议第 4 章引用第 6 章
- 同一公式在多个接口的「后端业务逻辑」中重复出现 → 建议集中到第 6 章

**简洁性**：
- 接口描述中的「后端业务逻辑」超过 5 个步骤且含冗余说明 → 标注可精简
- 表结构的「说明」列出现大段文字 → 建议一句话概括

**落地性**：
- 接口的「用途」描述不得使用「用于支持...」「用于满足...」等模糊表述
- 每个接口的业务逻辑步骤必须明确到表名、字段名、计算方式
- 每条规则必须能回答：**什么数据**、经过**什么处理**、产出**什么结果**

**充实度**：
- 第 8 章「开发注意事项」不得只写「无」→ 至少说明性能敏感点或依赖服务
- 第 2.5 章「空态/加载态」不得只写「显示空态」→ 必须有具体文案或组件

## 规则

- 需求覆盖是最高优先级——P0 功能点未覆盖属于严重问题，必须修复后才能开发
- **截图与需求分层原则**：UI 细节（布局/样式/格式）以截图为准，业务逻辑（功能点/规则/公式）以需求为准；仅 UI 细节差异不标记为严重问题，P0 功能点未覆盖仍为严重问题
- 前后端字段名不一致是严重问题（会导致联调失败）
- 审查报告必须同时输出到控制台和保存到文件
- 文档质量问题中的「重复内容」为一般问题（不阻断），但同一公式重复 3 次以上升级为严重
- 「水文/官话」性质的描述必须给出精简后的改写建议，不能只标注问题
- 空章节必须明确标注是「应删除此章节」还是「应补充内容」
- 审查完成后提示：`如设计文档已修复，运行 /gen-user-test-doc <功能名称> 生成测试文档，再运行 /dev/run <功能名称> 开始开发`
