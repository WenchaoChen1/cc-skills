# 生成产品功能需求文档

根据原始需求材料（文字描述、截图、草稿等），生成逻辑清晰的产品功能需求文档（PRD）。
文档面向产品经理视角，聚焦「功能是什么、规则是什么、公式是什么」。

> **严禁包含技术实现细节**：API 路径、HTTP 方法、数据库表/字段、Entity/DTO/Repository/Service/Controller 等代码层级术语、SQL 语句、前端组件库具体组件名。这些由 `/dev/gen-design-doc` 负责。

发现逻辑不闭环或表达不清晰时，自动提炼疑点问题，待用户回答后二次生成完整文档。

## 使用方式

```
/gen-requirement-doc <需求材料路径> [截图路径1] [截图路径2] ...
```

例如：
- `/gen-requirement-doc requirements/raw/财务看板草稿.docx screenshots/mockup.png`
- `/gen-requirement-doc requirements/raw/用户反馈.txt`

> 也可以不提供文件，直接在对话中粘贴需求描述，然后执行 `/gen-requirement-doc`

## 执行步骤

### 第〇步：确定功能名称与输出目录

1. 若 `$ARGUMENTS` 非空：从参数中提取文件路径（需求文档/截图），功能名称从第一个参数推断（`{features}/xxx/...` 取 `xxx`；其他文件取去掉路径和扩展名的文件名）
2. 若 `$ARGUMENTS` 为空：从对话内容中获取需求描述，并在处理完成后询问用户确认功能名称（小写 + 连字符）

**重复创建检查**（在生成文件前执行）：
- 若 `{features}/{name}/requirement/requirement-doc.md` 已存在：
  ```
  ⚠️ 文件已存在：{features}/{name}/requirement/requirement-doc.md
  建议先运行 /review-requirement-doc <name> 审查现有文档。
  选项：A. 覆盖（原文件被替换）  B. 取消（默认）  C. 查看现有文件内容
  ```
  等待用户选择，默认取消。
- 若不存在：正常创建目录并生成文件。

### 第一步：读取需求材料

**文件读取容错**：
- 文件路径不存在 → 提示「文件未找到：<路径>，请确认路径是否正确」，不继续后续步骤
- 文件存在但内容为空 → 提示「文件存在但内容为空，请检查文件内容」
- 编码问题（乱码） → 提示「文件编码异常，建议重新保存为 UTF-8」
- `.docx` 提取失败（缺少 python-docx） → 提示「需要安装 python-docx：`pip install python-docx`」

- 有文件路径：使用 Read 工具读取。`.docx` 文件用以下 Bash 命令提取全文：
  ```bash
  python -c "
  import sys
  sys.stdout.reconfigure(encoding='utf-8')
  from docx import Document
  doc = Document('<文件路径>')
  for para in doc.paragraphs:
      if para.text.strip():
          print(para.text)
  "
  ```
- 有截图：逐张读取图片，提取以下信息（用于第 8 章页面布局描述）：
  - 页面整体分区（导航栏 / 筛选区 / 主内容区 / 底部操作行等）
  - 表格/列表的列顺序、列宽比例、对齐方式
  - 层级关系（父行 / 子行 / 展开区域）
  - 固定元素的位置（左固定列、顶部固定行、悬浮按钮等）
  - 弹窗/抽屉的触发入口和内容区域
- 无参数：从当前对话中获取用户描述的需求内容；若描述中涉及页面结构，提取用于布局描述

### 第二步：提炼与整理

将原始材料整理为结构化内容，重点关注：

**功能边界**
- 这个功能做什么、不做什么
- 涉及哪些角色、哪些操作入口

**业务流程**
- 用户操作的完整路径（从触发到结果）
- 每个步骤的前置条件和后置结果
- 分支场景（条件A走流程X，条件B走流程Y）

**业务规则**
- 数据筛选、过滤、排序的规则
- 状态流转条件（什么情况下状态变化）
- 权限与可见性（谁能看、谁能操作）
- 数据为空/异常时的处理逻辑

**计算公式**
- 所有派生字段的计算方式
- 公式的适用范围（全量/按条件）
- 精度要求（小数位、取整方式）
- 分母为零等特殊情况的处理

**数据展示逻辑**
- 列表/表格展示哪些字段，顺序如何
- 汇总数据的统计口径
- 时间维度（按日/月/季/年）
- 数据来源范围（全部数据还是按条件过滤后的数据）

### 第三步：识别疑点，提炼问题

检查整理后的内容，识别以下类型的问题：

| 问题类型 | 判断标准 |
|---------|---------|
| **逻辑不闭环** | 有触发无结果、有条件无分支、流程走不到终态 |
| **规则不明确** | 表述模糊、有歧义、缺少边界条件 |
| **公式不完整** | 知道有计算但不知道具体公式或口径 |
| **范围不清晰** | 不确定数据统计口径、时间范围、适用角色 |
| **冲突矛盾** | 同一逻辑在不同地方描述不一致 |

**若发现疑点，进入「问题确认模式」：**

以如下格式展示问题，等待用户回答：

```
---
📋 在生成文档前，需要确认以下 N 个问题：

Q1. [问题类型] <具体问题>
    背景：<为什么需要确认这个问题>
    选项参考：A. <选项A>  B. <选项B>  C. 其他

Q2. [问题类型] <具体问题>
    背景：<为什么需要确认这个问题>

...

请回答以上问题后，我将生成完整文档。
若某些问题暂时无法确认，请注明「Q? 待定」，我会在文档中标注待确认。
---
```

**收到回答后**，**重新读取原始材料文件**（不依赖等待期前的记忆），将用户答案合并入重新读取的内容，继续执行第四步。

> ⚠️ 原因：用户回答问题前可能经过较长时间，之前读取的材料细节可能已从上下文中淡出。重新读取确保生成的文档基于完整信息，而非残缺记忆。

若原始材料逻辑完整、无疑点，直接进入第四步。

### 第四步：生成产品功能需求文档

按下方模板生成文档，**包含第 8 章页面层级说明**（逐页描述每个页面的作用、功能模块和层级关系）。

文档生成后、进入用户确认前，以"如果我是审查者，拿到这份文档会问什么"的视角快速扫一遍：

- **公式能跑通吗？** 对每个有除法的公式，想一下分母为零会怎样；对涉及排名/百分位的指标，用一组极端值走一遍计算链路，确认方向与排序的数学结果符合业务语义
- **同一件事只说了一遍吗？** 如果同一条规则/默认值/公式在多个章节出现，保留权威章节的定义（规则→§5，公式→§6，展示→§7），其他位置改为引用。两处说法不一致是最容易产生 bug 的问题
- **自造术语有定义吗？** 非行业通用的术语（如"正向排序"）在首次出现处必须有精确定义，且全文含义一致
- **公式变量都能追溯吗？** 公式中引用的子指标/变量，在本文档中有定义或注明了外部来源
- **引用是否注明来源？** 文中引用其他章节、外部文档、截图或沟通结论时，必须注明具体来源（如「见 §5.2」「来源：截图 mockup.png」「来源：2024-03-01 产品沟通结论」）
- **逻辑来源是否可追溯？** 凡描述中出现"按照XX规则"、"与XX一致"、"沿用XX逻辑"、"参照XX处理"等引用性逻辑语句，必须明确引用具体位置（如「校验规则见 §5.1」「与报表模块的筛选逻辑一致，见 §5.2 规则 3」）。如果逻辑是本文档首次定义的，不需要引用，但必须完整描述规则内容，不能用模糊措辞代替
- **数据查询是否说明数据源？** 涉及数据查询/展示的描述，必须明确数据从哪个业务模块获取、经过什么筛选条件，不能只写"系统数据"或"相关数据"
- **数据存入是否说明操作？** 涉及数据写入/更新的流程步骤，必须明确什么操作触发存入、存入什么数据、对已有数据做新增还是覆盖
- **重复描述是否已合并？** 同一规则/公式/默认值若在多处出现，是否已保留一处权威定义、其他位置改为引用格式（如「规则见 §5.1」）

发现问题时直接修正文档。无法自行决定的问题作为新的确认项展示给用户，等用户回复后再保存。

### 第五步：审查 UI 合理性，等待用户确认

文档生成后，对第 8 章的页面设计逐项检查，识别以下类型的潜在问题：

| 问题类型 | 判断标准 |
|---------|---------|
| **页面职责过重** | 单个页面承载了太多不相关的功能模块，应拆分 |
| **导航层级不合理** | 用户需要多次跳转才能完成一个核心操作 |
| **交互前后矛盾** | 同类操作（如编辑）在不同页面用了不同交互方式（一处弹窗、一处跳页） |
| **入口缺失** | 存在某个功能但找不到合理的页面入口 |
| **操作不可逆但无确认** | 删除/提交等不可逆操作没有确认机制 |
| **空状态缺失** | 页面在无数据时没有说明如何呈现 |
| **角色边界模糊** | 某个页面对不同角色的可见性或操作权限未说明 |

**若发现问题，以如下格式展示，等待用户逐条确认后再保存：**

```
---
🔍 在保存文档前，发现以下 N 处页面设计值得确认：

P1. [问题类型] <具体描述>
    位置：第 8 章 → <页面名称>
    说明：<为什么感觉不合理>
    建议：<可能的改进方向>
    → 请选择：A. 按原样保留  B. 采纳建议修改  C. 标注待确认

P2. [问题类型] <具体描述>
    ...

请逐条回复（如：P1:A P2:B P3:C），或说明你的想法。
---
```

**收到用户回复后**：
- **A（保留）**：该问题不写入文档，直接保存
- **B（修改）**：按建议调整第 8 章对应内容后保存
- **C（标注）**：在文档第 8 章对应位置标注 `⚠️ 待确认：<问题描述>`，并加入第 9 章待确认问题表

若没有发现任何问题，直接进入第六步整理格式。

### 第六步：整理文档格式与最终审查

在保存前，对文档进行最终格式整理和质量审查（本步骤侧重格式规范与统一性，是对第四步逻辑自审结果的最终确认）：

**格式整理：**
1. **合并重复定义**：扫描全文，将在多处重复描述的规则、公式、默认值合并到权威章节（规则→§5，公式→§6，展示→§7），其他位置替换为引用格式「见 §X.X」
2. **补齐引用来源**：检查所有引用（章节引用、外部来源、截图信息、沟通结论），确保每处引用都标注了具体来源
3. **补齐数据源说明**：检查 §7 中每个数据字段的「数据来源」描述，确保写明了具体的业务模块/数据维度/筛选条件，而非泛泛的"系统数据"
4. **补齐存入操作说明**：检查 §4 中涉及数据写入的步骤，确保每个存入操作都说明了触发条件、存入内容、操作类型（新增/更新/覆盖）
5. **统一格式规范**：
   - 表格列宽对齐、单元格内容风格一致
   - 标题层级正确（不跳级）
   - 公式格式统一（全用代码块或全用数学表达式）
   - 列表项句尾标点一致

**最终审查：**
- 从头到尾通读一遍，确认文档读起来流畅、无明显遗漏
- 确认目录结构与实际章节一一对应
- 确认所有 `⚠️ 待确认` 标注均已收录到 §9 待确认问题表

### 第七步：保存文档

**保存路径**：`<当前项目根目录>/{features}/{name}/requirement/requirement-doc.md`
（若目录不存在先创建 `{features}/{name}/requirement/` 文件夹）

功能名称从参数或内容中推断：若用户提供了文件路径，取去掉路径和扩展名后的文件名；若直接粘贴描述，根据功能主题生成英文短名（小写 + 连字符）。

保存后提示：
```
文档已保存至 {features}/{name}/requirement/requirement-doc.md
下一步：运行 /review-requirement-doc <功能名称> 审查需求，或 /dev/gen-design-doc <功能名称> 直接生成设计文档。
```

### 6.1 指标定义清单

> 每个指标按以下模板定义。根据指标的业务含义，选择需要明确的属性——不是每个指标都需要所有属性，但每个属性如果相关就必须写清楚。

根据指标特性，按需补充以下属性（思考维度，非强制全填）：

| 属性 | 何时需要 | 示例 |
|------|---------|------|
| 方向 | 涉及排名/百分位/颜色分级时 | 正向（越大越好）/ 反向（越小越好） |
| 排序 | 涉及排名/百分位计算时 | 升序/降序，需与方向在数学上自洽 |

| 除零处理 | 公式有除法时 | 分母为 0 时显示 N/A，不参与汇总 |
| N/A 处理 | 数据可能缺失时 | 展示灰色，不计入平均 |
| 子指标来源 | 公式引用了其他计算指标时 | "MRR YoY Growth Rate 定义见 §6.1.2" |

## 规则

### 核心原则

- **产品视角，不写实现**：不出现接口路径、数据库表名、技术框架等内容
- **每句话都能落地**：开发者读完每条规则，能直接写代码而不需要追问。避免"正常情况下"、"一般来说"、"适当"等无法转化为代码的表述
- **同一件事只说一遍**：规则在 §5 定义，公式在 §6 定义，展示在 §7 定义。其他章节引用而非重新描述。同一规则写两遍就会有两个版本，就会有 bug
- **公式必须经得起极端值**：有除法就想分母为零，有百分比就想负数和超 100%，有排名就验证方向与排序的数学一致性
- **疑点必须提问**：宁可多问也不擅自假设，但问题要精准，给出背景和参考选项
- **待定项必须标注**：用户回答「待定」的问题，在文档对应位置标注 `⚠️ 待确认：<问题编号>`

### 写作纪律

- **流程必须闭环**：每条路径有终态，每个"如果"有"否则"
- **术语必须定义**：非行业通用的术语首次出现时给出精确定义，全文含义一致
- **布局描述诚实**：有截图就基于截图描述；无视觉参考时标注「⚠️ 布局待设计」，不捏造
- **列/行结构要具体**：表格类页面列出每一列的名称、内容类型和排列顺序
- **引用必须注明来源**：引用本文档其他章节写明「见 §X.X」；引用其他功能模块写明「见 {features}/<模块名>/requirement/requirement-doc.md §X.X」；引用截图写明「来源：<截图文件名>」；引用沟通结论写明「来源：<日期> <沟通方式>」；引用行业标准/法规写明「来源：<标准/法规名称> <条款>」；引用其他外部文档写明「来源：<文档名称> <章节/页码>」
- **逻辑必须可追溯**：出现"按照XX规则"、"与XX一致"、"沿用XX方案"等引用性语句时，必须注明引用的具体章节或来源；如果是本文档首次定义的逻辑，必须完整写出规则内容，不能用"按既定规则"、"按之前的方案"等模糊表述代替
- **查询必须说明数据源**：描述数据展示/查询时，必须说明数据从哪个业务模块获取、经过什么筛选条件
- **存入必须说明操作**：描述数据写入时，必须说明什么操作触发存入、存入什么数据、是新增还是更新还是覆盖
- **重复内容必须合并**：同一规则/公式/默认值只在权威章节定义一次，其他位置用引用格式替代完整描述
