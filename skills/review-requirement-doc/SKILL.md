# 审查产品功能需求文档

站在"开发能否写代码、QA 能否写用例、PM 能否签字"三个角色视角，深度审查需求文档，输出按严重度排序的问题清单。

## 使用方式

```
/review-requirement-doc <功能名称 或 文档路径>
```

例如：
- `/review-requirement-doc financial-dashboard` — 按功能名称，自动读取 `{features}/financial-dashboard/requirement/` 目录
- `/review-requirement-doc 财务看板`
- `/review-requirement-doc path/to/my-requirement.md` — 直接指定文档路径，跳过固定目录扫描
- `/review-requirement-doc docs/需求v2.md screenshots/page1.png` — 指定文档 + 截图

> 传入功能名称时，对应 `{features}/{name}/` 文件夹，自动读取 `requirement/` 目录下所有文件
> 传入文件路径时，直接读取指定文件，不依赖 `{features}/` 目录结构

## 执行步骤

### 第一步：定位并读取所有材料

1. 若 `$ARGUMENTS` 非空，先判断参数类型：
   - **情况 A — 指定了文档路径**（参数含文件扩展名如 `.md`、`.docx`、`.txt`，或指向一个已存在的文件路径）：
     - 直接读取指定的文件作为待审查文档，**不扫描 `{features}/` 固定目录**
     - 支持同时传入多个文件路径（空格分隔），依次读取；若含截图路径一并读取
     - 审查报告保存路径：保存到文档所在目录的同级 `reviews/` 下（如 `path/to/reviews/requirement-review.md`）；若文档在项目根目录或无明确归属目录，保存到 `reviews/requirement-review.md`
     - **跳过下方第 3、4 条的目录扫描逻辑**
   - **情况 B — 仅指定功能名称**（参数不含文件扩展名，如 `xxx` 或 `{features}/xxx`）：
     - 解析功能名称（支持 `{features}/xxx` 或直接 `xxx` 两种写法），定位 `{features}/{name}/requirement/requirement-doc.md`
     - 继续执行下方第 3、4 条
2. 若 `$ARGUMENTS` 为空：
   - 扫描 `{features}/` 目录，列出所有子目录
   - 若只有一个：提示「检测到 `{features}/{name}/`，是否审查该功能？[Y/n]」
   - 若有多个：列出编号清单，提示用户选择
   - 若 `{features}/` 不存在或为空：提示「未找到功能目录，请提供功能名称，例如：/review-requirement-doc my-feature」
3. **仅情况 B 和空参数时执行**：读取 `{features}/{name}/requirement/` 目录下所有文件（逐一读取），包括截图
4. **仅情况 B 和空参数时执行**：若 `{features}/{name}/` 下有原始需求子目录，也扫描其中的截图并读取

**文件读取容错**：
- 目录或文件不存在 → 提示「缺少需求文档，请先运行 `/gen-requirement-doc <功能名称>`」，不继续后续步骤
- 文件存在但内容为空 → 提示「文件存在但内容为空，请检查 `{features}/{name}/requirement/requirement-doc.md`」
- 编码问题（乱码） → 提示「文件编码异常，建议重新保存为 UTF-8」

### 第二步：理解业务，建立心智模型

**先不找问题。先读懂文档在说什么。**

通读全文后，用自己的话总结：
- 这个功能解决什么业务问题？
- 核心用户是谁？核心操作路径是什么？
- 有哪些关键概念/指标/状态？它们之间如何关联？

> 这一步的理解会写入报告的「功能理解」章节（3-5 句话），既证明审查者读懂了文档，也帮助作者确认"你理解的是否就是我想表达的"。

### 第三步：多维度审查

> 以下维度是**思考透镜**，不是逐项打勾的检查表。根据文档类型灵活侧重——公式密集型文档重点看维度 B，流程密集型重点看维度 A，UI 密集型重点看维度 C。每个维度下的示例是启发性的，不是穷举。

#### 维度 A：业务逻辑完备性

**核心问题：开发拿到这份文档，能否不问任何人就写出代码？**

思考方向：
- **流程有没有走不到头的路径？** 每个分支都有终态吗？"如果条件 A"有没有对应的"否则"？
- **规则有没有写一半的？** 比如说了"按条件过滤"但没说条件是什么；说了"排序"但没说升序还是降序
- **同一个概念在不同地方是否矛盾？** 提取文档中的关键术语/规则/默认值，检查每处出现时描述是否一致。矛盾是最严重的问题——开发不知道该信哪个
- **版本范围是否明确？** 如果文档描述了一些"目前不支持"的功能，当前版本到底做哪些？
- **数据查询来源是否明确？** 涉及数据展示/查询的描述，是否说明了数据从哪个业务模块获取、经过什么筛选条件？模糊的"系统数据"或"相关数据"不可接受
- **数据存入操作是否明确？** 涉及数据写入/更新的流程步骤，是否说明了什么操作触发存入、存入什么数据、是新增还是更新还是覆盖？
- **逻辑引用是否可追溯？** 文中出现"按照XX规则"、"与XX一致"、"沿用XX方案"、"参照XX处理"等引用性逻辑语句时，是否注明了引用的具体章节或外部来源？如果是本文档首次定义的逻辑，是否完整描述了规则内容而非用模糊措辞代替？**未注明来源的逻辑引用必须逐条列入审查报告**

#### 维度 B：公式与计算逻辑

**核心问题：每个公式能否带入具体数字跑通？极端情况结果合理吗？**

> 若文档不涉及公式/计算，跳过此维度。

思考方向：
- **公式中的每个变量是否都有定义或来源说明？** 如果公式引用了其他指标，那些指标在哪里定义的？
- **边界值代入是否合理？** 分母为零、值为负数、值为零、值极大——代入后公式输出什么？文档有没有说明？
- **方向与排序是否自洽？** 如果涉及排名/百分位，"指标方向"（大好还是小好）和"排序方式"（升降序）在数学上是否一致？用极值代入走一遍"原始值→排序→排名→百分位"链路验证
- **精度是否明确？** 保留几位小数？怎么取整？不同场景是否一致？
- **汇总逻辑是否完整？** 缺失值怎么处理——排除？按零计？分母减一？

#### 维度 C：UI 与文档对照

**核心问题：UI 截图展示的页面结构和交互行为，与文档描述是否匹配？**

> 若无截图，跳过此维度。
>
> ⚠️ **重要前提**：UI 截图中的具体数值（如百分位 67%、分数 51%）是示例数据，不代表真实值。**绝不对比具体数字**。只对比结构、组件、行为。

思考方向：
- **页面结构**：截图中的区域划分（导航/筛选/内容/底部）与文档描述的页面布局是否一致？
- **组件存在性**：截图中可见的按钮、开关、筛选器、图表，文档是否都有描述？反过来，文档描述的组件截图中是否都能找到？
- **交互行为**：截图中暗示的交互模式（如切换按钮、展开/收起、多选/单选）与文档描述是否一致？
- **命名**：截图中的文字标签与文档中的术语是否一致？
- **默认状态**：截图展示的选中/展开/折叠状态与文档描述的初始状态是否一致？

#### 维度 D：可测试性

**核心问题：QA 能否根据这份文档写出"输入 X → 预期结果 Y"的测试用例？**

思考方向：
- **有没有模糊表述？** 如"合适的"、"正常情况下"、"一般来说"、"尽量"、"较好的体验"——这些无法转化为测试断言
- **每条规则能否构造反例？** 如果不能，说明规则定义不够精确
- **边界条件是否覆盖？** 空数据、零条记录、单条记录、最大值——文档有没有说明这些情况下的预期行为
- **错误状态是否描述？** 网络失败、数据加载失败、权限不足——用户看到什么？

#### 维度 E：文档结构与质量

**核心问题：文档结构是否完整、内容是否简洁无重复？**

思考方向：
- **关键章节是否覆盖？** 快速扫一遍文档是否涵盖以下内容（不要求标题完全匹配，按内容判断）：功能概述与范围、使用场景、功能清单与优先级、业务流程、业务规则、计算公式（如有）、数据展示、页面说明、待确认问题。缺失的章节在报告中标注。
- **同一规则是否在多处重复定义？** 应只在一处定义，其他地方引用
- **有没有"正确的废话"？** 如"提升用户体验"、"支持灵活配置"、"满足多种场景"——没有具体措施的描述应删除或具体化
- **章节是否有实质内容？** 仅有标题或"待补充"的空章节应标注
- **是否存在格式/编码错误？** 如 Excel 转换后的 `#NAME?`、乱码、断裂的表格
- **非通用术语是否有定义？** 非行业通用的术语在首次出现处是否给出了精确定义？全文中同一术语的含义是否一致？
- **是否混入技术实现细节？** 需求文档中不应出现 API 路径、数据库表名、HTTP 方法、Entity/DTO/Repository 等代码层级术语。若发现，标记为问题并建议移至设计文档
- **待确认项是否汇总完整？** 文档正文中所有 `⚠️ 待确认` 标注是否全部收录到 §9 待确认问题表？是否存在散落在正文中但未汇总的待确认项？
- **引用是否注明来源？** 文中引用其他章节、外部文档、截图或沟通结论时，是否标注了具体来源（如「见 §5.2」「来源：截图 mockup.png」）？无来源的引用导致信息不可追溯
- **重复定义是否已合并？** 同一规则/公式/默认值是否只在一处定义？若在多处重复描述，标记为问题并建议合并到权威章节、其他位置改为引用
- **文档格式是否统一？** 表格列宽对齐、标题层级不跳级、公式格式一致、列表标点风格一致

## 审查原则

- **先理解再审查**：不理解业务就开始找问题，只能找到表面问题。花时间建立心智模型，才能发现逻辑层面的矛盾
- **矛盾比缺失更严重**：同一件事在两处说法不同，比某个细节没写更危险——开发不知道该信哪个
- **每个问题必须有原文**：引用具体章节和原文，不泛泛地说"规则不够完善"
- **建议必须具体**：不说"建议补充边界条件"，而是说"ARR_(t-1)=0 时建议显示 N/A 并排除出百分位计算"
- **UI 截图是示例**：截图中的数值是样例数据，绝不拿来与公式反算对比。只对比结构、组件和交互行为
- **维度灵活运用**：维度侧重说明见第三步开头
- **引用和数据追溯见维度 A/E**：逻辑引用追溯（维度 A）和文档引用追溯（维度 E）分别检查，结果汇入追溯性检查清单
- **报告按严重度排序**：阻断性问题在前，改进建议在后。读者先看最重要的
- 审查完成后提示：`如需生成设计文档，运行 /dev/gen-design-doc <功能名称>`
