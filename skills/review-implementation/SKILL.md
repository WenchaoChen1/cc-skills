> **路径变量**：本 skill 使用 `config/defaults.json` 定义的路径变量。`{features}` 默认为 `cc-cache-doc/features`。详见 `config/README.md`。

# 审查功能实现闭环

读取 `{features}/{name}/` 下的需求文档、开发设计文档和测试文档，对照三份文档逐项扫描实际代码实现是否闭环，输出带 PASS/FAIL 状态的检查清单。

## 使用方式

```
/dev/review-implementation <功能名称 或 文档路径>
```

例如：
- `/dev/review-implementation financial-dashboard` — 按功能名称，自动读取 `{features}/` 下对应目录
- `/dev/review-implementation 财务看板`
- `/dev/review-implementation path/to/design-doc.md` — 直接指定设计文档路径
- `/dev/review-implementation design.md requirement.md test.md` — 同时指定多个文档

> 传入功能名称时，自动读取 `{features}/{name}/requirement/`、`{features}/{name}/dev-design/`、`{features}/{name}/user-test/` 目录
> 传入文件路径时，直接读取指定文件作为检查基准，不依赖 `{features}/` 目录结构

---

## 并行策略（Agent Teams）

> 当可以启动多个并行 Agent 时（TaskCreate / Agent tool 可用），后端核对和前端核对可以同时执行。若不可用，跳过本节，按下方「执行步骤」顺序执行。

### 并行执行流程

```
主线程（准备阶段）
  ↓ 执行第一步（读取基准材料）+ 第二步（提取检查清单）
  ├──→ Agent 后端：执行第三步后端探索 + 第四步 4-A、4-F 后端部分
  └──→ Agent 前端：执行第三步前端探索 + 第四步 4-B、4-C、4-D、4-E、4-F 前端部分、4-G
  ↓ 等待两个 Agent 均完成
主线程（合并阶段）
  ↓ 合并核对表 → 生成汇总报告（第五步）→ 保存文件（第六步）
```

### Agent 后端 — 后端审查

| 项目 | 说明 |
|------|------|
| **任务** | 探索后端实际代码（3 级定位策略）+ 核对 4-A（后端接口：路由、响应字段、参数校验、异常处理）+ 4-F 后端部分（异常场景的后端错误码） |
| **必须接收** | ① 设计文档第 4 章（接口汇总）、第 5 章（数据模型）、第 7 章（异常处理）的提取清单 ② user-test-doc.md 中与后端相关的验收条件（必填校验、唯一性约束，若存在） ③ CLAUDE.md 后端目录结构 ④ 后端 3 级定位策略说明 |
| **输出** | 后端接口核对表 + 异常处理后端核对表（格式同第五步报告模板中的对应表格） |

### Agent 前端 — 前端审查

| 项目 | 说明 |
|------|------|
| **任务** | 探索前端实际代码 + 核对 4-B（前端文件）、4-C（交互行为）、4-D（数据展示规则）、4-E（状态处理）、4-F 前端部分（前端错误提示）、4-G（测试文档验收） |
| **必须接收** | ① 设计文档第 2 章（UI）、第 3 章（前端实现）、第 7 章（异常处理）的提取清单 ② user-test-doc.md 完整验收条件（若存在） ③ CLAUDE.md 前端目录结构 ④ 路由配置文件路径 |
| **输出** | 前端文件核对表 + 交互行为核对表 + 数据展示核对表 + 状态处理核对表 + 异常处理前端核对表 + 测试文档验收核对表 |

### 主线程合并阶段

1. 等待两个 Agent 均完成
2. 合并所有核对表为统一审查报告格式（第五步模板）
3. 计算汇总统计（各类别 ✅/⚠️/❌ 数量）
4. 生成整体评估结论（完整闭环 / 存在缺口 / 严重缺口）和问题清单
5. 保存至 `{features}/{name}/reviews/implementation-review.md`（第六步）

## 执行步骤

### 第一步：读取所有基准材料

1. 若 `$ARGUMENTS` 非空，先判断参数类型：
   - **情况 A — 指定了文档路径**（参数含文件扩展名如 `.md`、`.docx`、`.txt`，或指向已存在的文件路径）：
     - 直接读取指定文件作为检查基准，**不扫描 `{features}/` 固定目录**
     - 支持多个文件路径（空格分隔），依次读取
     - 审查报告保存到文档所在目录的同级 `reviews/` 下
     - **跳过下方第 3 条的目录读取逻辑**
   - **情况 B — 仅指定功能名称**（参数不含文件扩展名，如 `xxx` 或 `{features}/xxx`）：
     - 解析功能名称，继续执行下方第 3 条
2. 若 `$ARGUMENTS` 为空：
   - 扫描 `{features}/` 目录，列出含 `dev-design/` 子目录的功能目录
   - 若只有一个：提示「检测到 `{features}/{name}/`，是否审查该功能？[Y/n]」
   - 若有多个：列出编号清单，提示用户选择
   - 若无匹配：提示「未找到设计文档，请先运行 /dev/gen-design-doc」
3. **仅情况 B 和空参数时执行** — 读取以下目录下所有文件（逐一读取）：
   - `{features}/{name}/dev-design/`（必须存在，作为核心检查基准）
   - `{features}/{name}/requirement/`（若存在，提取业务规则和验收标准）
   - `{features}/{name}/user-test/`（若存在，提取功能测试用例和验收条件，补充边界场景）
4. 读取 `CLAUDE.md`，了解项目技术栈和目录结构

**文件读取容错**：
- `dev-design/` 目录不存在 → 提示「缺少设计文档，请先运行 `/dev/gen-design-doc <功能名称>`」，不继续后续步骤
- `requirement/` 或 `user-test/` 目录不存在 → 跳过对应检查维度，在报告中标注「无需求文档/测试文档，相关维度未检查」
- 文件存在但内容为空 → 提示「文件存在但内容为空，请检查对应文件」
- 编码问题（乱码） → 提示「文件编码异常，建议重新保存为 UTF-8」

### 第二步：提取检查清单

从设计文档中提取以下待检查项：

**后端接口清单**（来自设计文档第 4.1 章接口汇总）
- 每个接口的：方法、路径、必要参数、响应字段

**前端文件清单**（来自设计文档第 3.1 章路由与文件结构）
- 页面文件路径、组件文件路径、路由配置

**交互行为清单**（来自设计文档第 2.4 章交互行为）
- 每条交互：触发元素、触发方式、预期效果

**异常处理清单**（来自设计文档第 7 章异常处理）
- 每个异常场景：后端错误码、前端展示方式

**数据展示规则**（来自设计文档第 2.3 章）
- 数字格式、颜色语义、空值处理

**验收条件清单**（来自 user-test-doc.md，若存在）
- 提取功能测试用例中的"预期结果"作为代码验收标准
- 提取边界测试用例中的边界条件（如必填校验、唯一性约束、上限限制等）
- 提取 UI 验收标准（加载态、空态、错误态、响应式要求）

### 第三步：探索实际代码

根据 CLAUDE.md 中的目录结构，定位并读取实际代码文件：

**后端 — 3 级定位策略**
1. **按接口路径前缀推断业务域目录**：根据 CLAUDE.md 中的模块结构，将接口路径前缀映射到对应的业务域目录
2. **按 Controller 注解搜索**：在后端项目源码目录下搜索 `@RequestMapping`/`@GetMapping`/`@PostMapping` 注解中包含接口路径的 Controller 文件
3. **按关键词搜索**：若前两步未找到，在后端项目整体范围内搜索接口路径或方法名关键词

**前端（React）**
- 按设计文档第 3.1 章的文件路径直接读取
- 读取路由配置文件，确认路由是否注册
- 读取 API service 文件，确认接口调用

### 第四步：逐项核对

#### 4-A. 后端接口核对

> **Agent Teams 模式**：4-A 和 4-F 后端部分由 Agent 后端并行执行（见「并行策略」章节）。

对每个接口逐项检查：
- [ ] Controller 中有对应的方法和路由注解
- [ ] 路径与设计文档完全一致（包括路径参数名）
- [ ] 响应中包含设计文档要求的所有字段
- [ ] 必填参数有校验（`@NotNull`/`@Valid` 或手动校验）
- [ ] 有对应的业务异常处理

#### 4-B. 前端文件核对

> **Agent Teams 模式**：4-B 至 4-E、4-F 前端部分、4-G 由 Agent 前端并行执行（见「并行策略」章节）。

- [ ] 页面文件存在（路径与设计文档一致）
- [ ] 路由已在路由配置文件中注册
- [ ] API service 函数存在（每个后端接口有对应的调用函数）
- [ ] TypeScript interface 字段名与后端响应字段名一致

#### 4-C. 交互行为核对

对设计文档第 2.4 章每条交互逐项检查代码中是否有对应实现：
- 点击事件 handler
- onChange 回调
- 弹窗/抽屉的打开/关闭逻辑

#### 4-D. 数据展示核对

- [ ] 数字格式化是否符合设计规范（小数位、货币符号）
- [ ] 正负数颜色是否按设计实现
- [ ] 空值/null 处理是否与设计一致（显示"--"还是"0"还是空）

#### 4-E. 状态处理核对

- [ ] 加载态（Loading state）是否实现
- [ ] 空态（无数据时）是否实现
- [ ] 错误态（接口报错时）是否实现

#### 4-F. 异常处理核对

对设计文档第 7 章每个异常场景：
- [ ] 后端是否返回对应的错误码
- [ ] 前端是否有对应的错误提示（Toast/Modal/内联提示）

#### 4-G. 测试文档验收核对（有 user-test-doc.md 时执行）

对 user-test-doc.md 中每条功能测试用例的"预期结果"：
- [ ] 代码中是否有对应的实现路径（正向逻辑）

对边界测试用例：
- [ ] 必填字段是否有校验（前端/后端均需）
- [ ] 唯一性约束是否在后端落地（数据库约束或业务校验）
- [ ] 数量上限/下限是否有对应的边界处理

对 UI 验收标准：
- [ ] 加载中、空态、错误态是否均有实现

## 规则

- 必须找到代码文件后再判断，不得凭猜测标注 ✅
- 若文件找不到，使用 Grep 搜索接口路径或组件名，不得直接标 ❌
- 后端接口路径不一致（大小写/斜杠差异）属于严重问题
- 前端 TypeScript 字段名与后端不一致属于严重问题
- 审查报告必须同时输出到控制台和保存到文件
- 审查完成后提示：`如需生成自动化测试代码，运行 /dev/gen-unit-test <功能名称>`
