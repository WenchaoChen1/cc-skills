# Asana Ticket 需求审查

审查 Asana ticket 的需求质量，找出问题并给出改进建议。

## 使用方式

```
/review-asana-ticket <一个或多个 Asana 链接>
```

## 输入参数

$ARGUMENTS — 一个或多个 Asana task 链接（URL 或 task GID），空格或换行分隔。

---

## 执行流程

### Phase 1: 解析输入

从 `$ARGUMENTS` 中提取所有 task 标识：
- URL 格式：从 `https://app.asana.com/.../task/<GID>` 中提取 GID
- 纯 GID 格式：直接使用
- 记录输入的 ticket 总数，标记为 `单票模式` 或 `多票模式`

### Phase 2: 信息采集（并行）

对每个 ticket **并行**发起 Agent 采集全量信息：

#### 2.1 基础信息

使用 `mcp__claude_ai_Asana__get_task` 获取：
- task_id: 提取的 GID
- include_comments: true
- include_subtasks: true
- comment_limit: 50
- opt_fields: `name,notes,html_notes,assignee.name,due_on,due_at,start_on,completed,completed_at,created_at,modified_at,custom_fields,dependencies,dependents,followers.name,parent.name,memberships.project.name,memberships.section.name,tags.name,permalink_url,num_subtasks`

从返回结果中记录：
- **描述**（notes / html_notes）
- **负责人**（assignee）
- **截止日期**（due_on）
- **子任务列表**（subtasks）
- **评论列表**（comments/stories）
- **依赖关系**（dependencies / dependents）
- **自定义字段**（custom_fields）
- **标签**（tags）
- **所属项目和 Section**（memberships）
- **关注者**（followers）

#### 2.2 递归子任务

对每个子任务，再次调用 `mcp__claude_ai_Asana__get_task` 获取其：
- 描述、子任务（二级）、评论
- 如果二级子任务还有子任务，继续递归（最多 3 层）

#### 2.3 附件

使用 `mcp__claude_ai_Asana__get_attachments` 获取：
- parent: task GID
- opt_fields: `name,download_url,permanent_url,view_url,resource_type,created_at`

记录附件数量和名称列表。

### Phase 3: 单票审查

对每个 ticket 独立执行以下 6 个维度审查，每个维度打分（1-5 分）并列出具体问题：

#### 3.1 需求完整性（Completeness）

检查项：
- [ ] **目标描述**：是否清晰说明了"做什么"和"为什么做"
- [ ] **用户故事**：是否包含 As a / I want / So that 或等效描述
- [ ] **验收标准（AC）**：是否有明确的、可验证的验收条件
- [ ] **边界条件**：是否说明了异常情况、错误处理、空状态
- [ ] **UI/UX 规格**：是否有设计稿、原型图或 UI 描述（或附件引用）
- [ ] **数据需求**：是否说明了涉及的数据模型、API 接口
- [ ] **非功能需求**：是否提及性能、安全、兼容性、国际化等

缺失项标记为 `MISSING`，不完整标记为 `INCOMPLETE`。

#### 3.2 需求清晰度（Clarity）

检查项：
- [ ] **无歧义**：描述是否存在多种理解方式
- [ ] **量化指标**：涉及性能/数量时是否有具体数字
- [ ] **范围界定**：是否明确了"不做什么"（out of scope）
- [ ] **术语一致**：使用的业务/技术术语是否前后一致
- [ ] **假设说明**：是否列出了前置假设

对每个歧义点，给出具体的歧义描述和建议的澄清问题。

#### 3.3 子任务分解（Decomposition）

检查项：
- [ ] **覆盖度**：子任务是否覆盖了需求描述中的所有工作
- [ ] **粒度合理**：每个子任务是否可在 1-2 天内完成
- [ ] **无遗漏**：是否缺少测试、文档、review 等常见子任务
- [ ] **依赖顺序**：子任务之间的执行顺序是否合理
- [ ] **负责人分配**：每个子任务是否有明确的负责人

列出建议新增的子任务和需要拆分的过大子任务。

#### 3.4 协作健康度（Collaboration）

从评论中分析：
- [ ] **未决问题**：是否有提问但未回答的评论
- [ ] **阻塞点**：是否有人报告被阻塞
- [ ] **决策记录**：重要决策是否在评论中有记录
- [ ] **讨论热度**：评论数量和最近活跃度
- [ ] **信息完整**：评论中提到的附件/链接是否可访问

列出所有未解决的问题，按重要性排序。

#### 3.5 风险评估（Risk）

评估以下风险：
- [ ] **范围蔓延**：需求描述是否过于模糊导致可能膨胀
- [ ] **技术风险**：是否涉及新技术、复杂集成、未验证方案
- [ ] **依赖风险**：外部依赖是否明确、是否有 fallback
- [ ] **时间风险**：工作量与截止日期是否匹配
- [ ] **人员风险**：负责人是否过载（同时分配多个任务）

每个风险标记等级：`LOW` / `MEDIUM` / `HIGH` / `CRITICAL`。

#### 3.6 可测试性（Testability）

检查项：
- [ ] **AC 可验证**：每条验收标准是否可以写成测试用例
- [ ] **测试数据**：是否说明了测试所需的数据或环境
- [ ] **回归影响**：是否说明了可能影响的现有功能

---

### Phase 4: 综合审查（仅多票模式）

当输入 >= 2 个 ticket 时，额外执行综合交叉审查：

#### 4.1 跨票依赖分析

- 识别票据之间的隐含依赖（同一模块、同一 API、同一数据表）
- 检查依赖关系是否在 Asana 中正确设置
- 建议最优执行顺序

#### 4.2 范围重叠检查

- 检查是否有多个 ticket 涉及同一功能区域
- 识别潜在冲突（同时修改同一文件/模块）
- 标记需要协调的工作

#### 4.3 资源分配分析

- 统计每个负责人的任务数
- 识别单点瓶颈（某人承担过多关键任务）
- 检查是否有未分配的任务

#### 4.4 优先级与排期

- 检查任务优先级是否合理
- 所在 Section（To Do / In Development 等）是否反映真实状态
- 截止日期是否合理分布

#### 4.5 一致性检查

- 命名规范是否统一
- 描述格式是否统一
- 标签使用是否一致
- 验收标准格式是否一致

#### 5.2 评分标准

- **5 分 (A)**：完整、清晰、无明显问题
- **4 分 (B)**：基本完整，有小的改进空间
- **3 分 (C)**：可用但有明显缺失，需要补充
- **2 分 (D)**：问题较多，需要大幅修改
- **1 分 (F)**：严重缺失或完全不可用

---

## 约束

- 所有审查基于 Asana 中的现有信息，不做超出 ticket 内容的假设
- 审查语言：中文，技术术语保留英文
- 对每个问题必须给出具体、可操作的改进建议，不做泛泛而谈
- 评论中的对话要完整阅读，理解上下文后再评判
- 子任务递归最多 3 层，避免过度展开
