# 快速修复管道

从 bug 描述出发，快速定位代码、分析根因、执行修复、运行测试、生成修复报告。
适用于紧急 bug 修复，不需要走完整 9 步工作流。

## 使用方式

```
/dev/hotfix <bug描述> [截图路径...] [--feature=<功能名>]
```

例如：
- `/dev/hotfix "财务看板金额显示为NaN" screenshots/bug.png`
- `/dev/hotfix "新增记录后列表未刷新" --feature=benchmark-entry`
- `/dev/hotfix "导出Excel时报500错误"`

**参数说明**：

| 参数 | 说明 |
|------|------|
| `<bug描述>` | bug 的文字描述（必填，用引号包裹） |
| `[截图路径...]` | bug 截图，可传多个，可选 |
| `--feature=<功能名>` | 指定关联的功能目录，用于缩小代码搜索范围，可选 |

---

## 执行步骤

### 第一步：理解 Bug

1. 读取 bug 描述和截图（若有）
2. 提取关键信息：
   - 哪个页面/功能出问题
   - 什么操作触发了问题
   - 期望行为 vs 实际行为
   - 错误信息（若有）
3. 读取 `CLAUDE.md` 了解项目结构和技术栈

### 第二步：定位代码

**搜索策略（逐级扩大范围）**：

1. **若指定了 `--feature`**：
   - 读取 `{features}/{name}/dev-design/dev-design-doc.md`，从设计文档中定位相关接口和页面文件
   - 直接读取对应的 Controller、Service、前端页面文件

2. **按错误信息搜索**：
   - 若有 HTTP 错误码 → 搜索对应 Controller 路由
   - 若有 JS 错误 → 搜索前端组件中的关键字
   - 若有 Java 异常堆栈 → 搜索异常类和调用链
   - 若有 SQL 错误 → 搜索 Repository/DAO 和 SQL 映射文件

3. **按功能关键词搜索**：
   - 从 bug 描述中提取关键词（如字段名、页面名、操作名）
   - 在后端代码中搜索（CLAUDE.md 中的后端项目目录）
   - 在前端代码中搜索（CLAUDE.md 中的前端项目目录）

4. **按路由/接口搜索**：
   - 若涉及具体页面 → 搜索路由配置，定位页面组件
   - 若涉及具体接口 → 搜索 `@RequestMapping`/`@GetMapping`/`@PostMapping` 注解

**输出定位结果**：
```
【代码定位】
▌ 相关文件（共 N 个）：
- <文件路径> — <与 bug 的关系>
- ...

▌ 初步判断：
- 问题可能出在 <文件>:<行号> 的 <方法/组件>
- 原因：<初步分析>
```

### 第三步：分析根因

读取定位到的代码文件，深入分析：

1. **追踪数据流**：从前端触发 → API 调用 → Service 处理 → 数据库查询，找到断裂点
2. **检查常见问题模式**：
   - 空指针/undefined 访问
   - 类型转换错误（如 String→Number、null 参与运算导致 NaN）
   - 条件判断遗漏（缺少 else 分支、switch 无 default）
   - 异步处理不当（Promise 未 await、竞态条件）
   - SQL 查询条件错误（缺少 WHERE 条件、JOIN 类型错误）
   - 前后端字段名不一致
   - 时区/日期格式处理错误
   - 缓存未失效
3. **确认根因**后输出分析：

```
【根因分析】
▌ 根本原因：<一句话描述>
▌ 详细说明：<技术细节>
▌ 影响范围：<哪些功能/页面会受影响>
▌ 涉及文件：
- <文件路径>:<行号> — <需要修改的内容>
```

### 第四步：生成修复方案（等待用户确认）

在执行修复前，必须向用户展示修复方案并获得确认：

```
【修复方案】

▌ 方案概述：<一句话描述修复思路>

▌ 修改清单（共 N 个文件）：
1. <文件路径>
   - 修改：<描述修改内容>
   - 原因：<为什么这样改>

2. <文件路径>
   - 修改：<描述修改内容>
   - 原因：<为什么这样改>

▌ 风险评估：
- 影响范围：<仅修复点 / 可能影响其他功能>
- 兼容性：<是否需要数据库变更 / 是否破坏现有接口>
- 回滚方案：<如何回滚此修复>

是否执行修复？[Y/n]
```

等待用户确认后执行第五步。

### 第五步：执行修复

1. **先读后写**：修改每个文件前，先完整读取文件内容
2. **最小化改动**：只修改必要的代码，不做额外重构
3. **保持风格一致**：修改的代码风格与周边代码一致
4. **不修改与 bug 无关的代码**

### 第六步：验证修复

1. **自动编译检查**：
   - 后端（若有 Java 修改）：
     ```bash
     cd <后端项目根目录> && mvn compile -pl <模块名> -q 2>&1
     ```
   - 前端（若有 TS/JS 修改）：
     ```bash
     cd <前端项目根目录> && npx tsc --noEmit 2>&1
     ```

2. **运行相关测试**（若存在）：
   - 搜索与修改文件相关的测试文件（`*Test.java`、`*.test.tsx`）
   - 后端：`mvn test -pl <module> -Dtest=<TestClass> -q`
   - 前端：`npm run test -- --testPathPattern="<pattern>" --watchAll=false`

3. **若编译/测试失败**：
   - 分析错误信息
   - 尝试修复（最多 2 轮）
   - 若仍失败 → 输出失败原因，等待用户介入：
     ```
     ⚠️ 修复后验证未通过：
     - <错误描述>
     选项：A. 我来手动修复  B. 回滚修改  C. 忽略继续
     ```

### 第七步：生成修复报告

输出修复报告至控制台，同时保存至文件：

```markdown
## Hotfix 修复报告

**Bug 描述**：<原始描述>
**修复时间**：<日期时间>
**关联功能**：<功能名 或 "无">

---

### 根因

<根因的一句话描述>

<详细技术说明>

### 修改文件

| 文件 | 修改类型 | 说明 |
|------|---------|------|
| `<路径>` | 修改/新增/删除 | <修改说明> |

### 具体改动

#### `<文件路径>`

<描述改动内容和改动前后的关键代码差异>

### 验证结果

| 检查项 | 状态 |
|--------|------|
| 编译通过 | ✅/❌ |
| 相关测试通过 | ✅/❌/⚠️ 无测试文件 |

### 影响范围

- <影响说明>

### 建议后续动作

- [ ] 运行完整测试套件确认无回归
- [ ] 若影响范围较大，建议运行 `/dev/review-implementation <feature>` 做一次全面审查
- [ ] 考虑补充单元测试覆盖此场景
- [ ] 建议 git commit message：`fix(<模块>): <修复描述>`
```

**保存路径**：
- 若指定了 `--feature`：`{features}/{name}/reviews/hotfix-<YYYY-MM-DD>.md`
- 若未指定：仅输出到控制台

---

## 规则

- 修复前**必须**向用户展示方案并获得确认，不得直接修改代码
- 不修改与 bug 无关的代码，不做"顺手"重构
- 若 bug 涉及数据库变更（DDL/DML），必须在方案中**单独标注并额外确认**
- 编译检查和测试运行失败时，最多自动重试 2 轮
- 若无法定位根因，诚实告知并给出排查方向（如需要加日志、需要查数据库数据等），不猜测性修改
- 修复报告必须包含具体的改动说明，方便 code review
- 若修复过程中发现其他隐患，在报告「建议后续动作」中列出，但不在本次修复中处理
