# 后端实现模板

> 通用后端实现流程，适用于所有后端语言（Java、Python、Go、Rust、Kotlin、C# 等）。
> 语言特定的编码规范从 `rules/{lang}/` 加载，本文件只定义实现步骤顺序。
> 公共规则见 `common.md`。

---

## 前置：从 CLAUDE.md 和现有代码确认项目信息

开始实现前，必须确认以下信息（**不得使用预设值**，全部从 CLAUDE.md 和现有代码获取）：

| 需确认项 | 获取方式 |
|---------|---------|
| 语言版本 | `CLAUDE.md` 或项目配置文件 |
| Web 框架 | `CLAUDE.md` 或现有代码 |
| ORM / 数据库访问方式 | 现有数据模型文件 |
| 数据验证 / DTO 方式 | 现有 Schema/DTO 文件 |
| 映射工具（若有） | 现有映射文件 |
| 异常处理方式 | 现有 Service/Controller |
| 统一响应格式 | 现有 Controller 返回类型 |
| 模块组织方式 | 项目目录结构 |
| 包命名 / 模块命名约定 | 现有代码 |
| 分层目录结构 | 现有代码（如 controller/service/repository 或其他命名） |
| 路由注册方式 | 现有代码 |
| 依赖管理工具 | 项目配置文件 |

> 不同语言的具体项（如 Java 的 Lombok、Python 的类型注解）通过 `rules/{lang}/coding-style.md` 获取。

## 实现步骤

按设计文档从底层到上层逐一实现。具体层名和文件组织以项目实际结构为准。

### 1-A. 数据库 DDL（若有新表）

- 按设计文档的表结构建表
- 加字段注释
- 创建必要索引
- 文件放在项目约定的 migration/SQL 目录

### 1-B. 数据模型层

- 位置：参照现有模型文件路径
- 每张新表对应一个模型类 / 结构体
- 遵循现有模型文件的注解、命名、继承风格
- 审计字段处理方式参照现有代码
- 若项目直接写 SQL 无 ORM → 跳过此步

### 1-C. 数据访问层

- 位置：参照现有 Repository/DAO 路径
- 继承项目现有的基类或接口
- 按设计文档的查询/排序需求添加方法
- 查询方式参照现有代码

### 1-D. 数据传输层

- 位置：参照现有 DTO/Schema/VO 路径
- 响应结构：含 id、所有业务字段、时间戳
- 请求结构：必填字段有校验标注
- 编辑请求：字段可选（PATCH 语义）
- 命名和校验方式参照现有代码

### 1-E. 映射层（若项目有此层）

- 参照现有映射文件的工具和风格
- 若项目无独立映射层 → 跳过

### 1-F. 业务逻辑层

- 位置：参照现有 Service 路径
- 接口/实现分离（若项目有此惯例）
- 写操作加事务（参照现有方式）
- 校验失败抛项目约定的异常
- 日志、分页等参照现有 Service

### 1-G. 接口层

- 路径与设计文档**完全一致**
- 使用项目约定的注解/装饰器
- 返回项目统一响应格式
- 入参校验方式参照现有代码

### 1-H. 路由注册（若需要）

- 在项目入口文件注册新模块路由
- 参照现有路由注册方式和顺序

### 1-I. 自检

完成后逐项确认：
- [ ] 所有接口路径与设计文档一致
- [ ] 所有字段名与设计文档一致
- [ ] 必填字段有校验
- [ ] 写操作有事务（若语言支持）
- [ ] 数据模型主键、审计字段与现有代码风格一致
- [ ] 无编译/语法错误
- [ ] 没有引入不必要的新依赖
- [ ] 路由已注册
- [ ] 遵循 `rules/{lang}/` 下的编码规范
