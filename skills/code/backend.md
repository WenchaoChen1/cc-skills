# 后端实现模板

> 通用后端实现流程，适用于所有后端语言。
> 语言特定编码规范从 `rules/{lang}/` 加载。
> 安全规范从 `rules/{lang}/security.md` 加载。
> 公共规则见 `common.md`。

## 前置：从 CLAUDE.md 和现有代码确认项目信息

**不得使用预设值**，全部从 CLAUDE.md 和现有代码获取：

| 需确认项 | 获取方式 |
|---------|---------|

## 实现步骤

从底层到上层逐一实现。具体层名和文件组织以项目实际结构为准。

### 1-A. 数据库 DDL（若有新表）

- 按设计文档的表结构建表
- 加字段注释
- 创建必要索引
- 文件放在项目约定的 migration/SQL 目录

### 1-B. 数据模型层

- 参照现有模型文件路径、注解、命名、继承风格
- 审计字段处理方式参照现有代码
- 若项目直接写 SQL 无 ORM → 跳过

### 1-C. 数据访问层

- 继承项目现有的基类或接口
- 按设计文档的查询/排序需求添加方法
- 查询方式参照现有代码

### 1-D. 数据传输层

- 响应结构：含 id、所有业务字段、时间戳
- 请求结构：必填字段有校验标注
- 命名和校验方式参照现有代码

### 1-E. 映射层（若项目有此层）

- 参照现有映射文件的工具和风格
- 若项目无独立映射层 → 跳过

### 1-F. 业务逻辑层

- 接口/实现分离（若项目有此惯例）
- 写操作加事务
- 校验失败抛项目约定的异常

### 1-G. 接口层

- 路径与设计文档**完全一致**
- 使用项目约定的注解/装饰器
- 返回项目统一响应格式
- 入参校验方式参照现有代码

### 1-H. 路由注册（若需要）

- 在项目入口文件注册新模块路由

### 1-I. 自检

> 除 common.md 通用检查项外，额外检查：

**功能检查**：
- [ ] 所有接口路径与设计文档一致
- [ ] 所有字段名与设计文档一致
- [ ] 必填字段有校验
- [ ] 写操作有事务
- [ ] 数据模型主键、审计字段与现有代码风格一致
- [ ] 路由已注册
- [ ] 遵循 `rules/{lang}/` 下的编码规范

**安全检查**：
- [ ] 无硬编码密钥（搜索 API key、password、secret、token 字面量）
- [ ] 用户输入有参数校验（类型、长度、范围）
- [ ] 数据库查询使用参数化方式（无字符串拼接 SQL）
- [ ] 错误响应不暴露堆栈、SQL、内部路径
- [ ] 写操作接口有认证/授权检查（参照现有认证方式）
- [ ] 敏感操作有日志记录（日志中不含密码/token）
- [ ] 遵循 `rules/{lang}/security.md`
