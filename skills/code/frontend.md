## 实现步骤

### 2-A. API Service 层

- 位置：参照现有 service 文件的目录和命名
- 为设计文档每个接口写一个请求函数
- 使用项目现有的请求工具和封装方式
- **字段名必须与后端响应结构完全一致**

### 2-B. 页面组件

**实现时逐条对照设计文档**：

| 设计文档章节 | 实现内容 |
|-------------|---------|
| 页面布局 | DOM/组件结构，间距比例 |
| 组件清单 | 逐一实现每个组件 |
| 数据展示规则 | 数字格式化、颜色语义、空值显示 |
| 交互行为 | 每条交互逐一实现，**不遗漏** |
| 空态/加载态 | Loading、空数据提示、错误态 |

- 页面入口组件负责数据加载和状态管理
- 子组件不直接调用 API（由父组件传入数据和回调）
- 样式使用项目现有方案

### 2-C. 路由注册

- 在路由配置文件注册新路由
- 权限字段参照同类已有路由

### 2-D. 自检

- [ ] 类型定义字段名与后端一一对应
- [ ] 所有 API 调用路径与后端接口路径一致
- [ ] 每条交互行为都有对应实现
- [ ] 空态和加载态已实现
- [ ] 路由已注册
- [ ] 数字格式化规则一致
- [ ] 遵循 `rules/typescript/` 和 `rules/web/` 的编码规范

## 常见错误预防

| 错误类型 | 预防措施 |
|---------|---------|
| 类型字段名与后端不一致 | 逐字段核对后端数据存档 |
| 表格 colSpan/rowSpan 计算错误 | 行计数函数覆盖所有行类型 |
| 路由权限配置报错 | 参照同类已有路由 |
| 数字格式化不一致 | 先搜索项目是否已有格式化工具函数 |
| 组件 key 缺失 | 列表渲染使用唯一且稳定的 key |
