# 写代码 — 统一代码编写

动态发现规则，TDD 驱动开发，写完自动验证和规则合规审查。

> **子文件**：
> - `common.md` — 公共开发规范 + 安全底线（始终加载）
> - `backend.md` — 通用后端实现模板
> - `frontend.md` — 前端实现模板

## 使用方式

```
/code <功能名称或描述>
```

- **完整模式**：有设计文档时，按设计文档逐章实现
- **轻量模式**：无设计文档时，根据用户描述 + CLAUDE.md + 现有代码直接开发

---

## 执行流程

```
第一步：准备（配置 + 规则 + 材料）→ 输出规则报告
    ↓
第二步：上下文构建（技术栈检测 + 代码探索）→ 歧义时暂停
    ↓
第三步：TDD 开发（测试先行 → 实现 → 重构 → Git 检查点）
    ↓
第四步：验证循环（编译 → Lint → 测试 → 安全 → 交叉验证）
    ↓
第五步：规则合规审查 → 最终报告
```

---

## 第一步：准备

> 静默执行，末尾输出规则报告。

### 1-A. 加载路径配置

从 `cc-skills.json` 读取路径变量。详见 `config/README.md`。

### 1-B. 发现并加载规则

按优先级扫描，同名高优先级覆盖低优先级：

| 优先级 | 位置 |
|--------|------|
| 1（最高） | `{standards}` — 项目规范 |
| 2 | `{rules}` — 项目规则 |
| 3 | `{personal_rules}` — 个人规则 |
| 4（最低） | 本插件 `rules/` — 插件默认 |

扫描各目录中的子目录，记录可用技术栈规则。
始终加载 `rules/common/` + 本目录 `common.md`。

**不硬编码支持哪些语言**——有规则目录就支持。

### 1-C. 读取材料

- 读 CLAUDE.md（技术栈、目录结构、构建/测试命令）
- 读 `{standards}` 下的代码规范
- 检查 `{features}/{name}/dev-design/`：存在 → **完整模式**；不存在 → **轻量模式**
- 完整模式下读需求文档（若存在）

### 1-D. 输出规则报告

```
【规则加载报告】
扫描目录：{standards} / {rules} / {personal_rules} / 插件 rules/
发现的技术栈规则：java, python, typescript, web, ...
安全规则：已加载
测试规则：已加载（rules/{lang}/testing.md）
模式：完整 / 轻量
构建命令：（从 CLAUDE.md 提取）
测试命令：（从 CLAUDE.md 提取）
```

## 第二步：上下文构建

> 静默执行，歧义时暂停。

### 2-A. 检测技术栈

1. 从 CLAUDE.md 识别项目技术栈
2. 从设计文档或用户描述识别本次涉及的技术栈
3. 与可用规则取交集

**明确时自动继续，有歧义时暂停确认。**

### 2-B. 探索现有代码模式

对每种技术栈：
- 读取编码规范、安全规范、测试规范
- 找到至少 2 个同类现有文件作为模式参考
- 找到至少 1 个同类现有测试文件作为测试模式参考

---

## 第三步：TDD 开发

> **铁律：没有失败的测试，不写实现代码。**
> 每个功能点/接口按 RED → GREEN → REFACTOR 循环。

### 3-A. RED — 先写失败的测试

1. 根据设计文档（完整模式）或用户描述（轻量模式），为每个功能点写测试
2. 测试框架和组织方式参照项目现有测试 + `rules/{lang}/testing.md`
3. **运行测试，确认失败**
   - 失败原因必须是"功能缺失"，不是语法错误
   - 如果测试直接通过 → 测试写错了，修改测试

### 3-B. GREEN — 写最小实现

1. 路由到 `backend.md` 或 `frontend.md` 执行实现步骤
2. 只写刚好通过测试的代码，不多写
3. **运行测试，确认通过**
   - 所有测试必须通过（新测试 + 已有测试）
   - 如果已有测试被破坏 → 修复，不能跳过

### 3-C. REFACTOR — 重构

1. 在保持测试全绿的前提下改进代码
2. 移除重复、改进命名、提取公共逻辑
3. **运行测试，确认仍然通过**

### 3-D. Git 检查点

每个 RED → GREEN → REFACTOR 循环完成后提交：

```bash
# RED 验证后
git commit -m "test: add tests for <功能点>"

# GREEN 验证后
git commit -m "feat: implement <功能点>"

# REFACTOR 后（若有改动）
git commit -m "refactor: clean up <功能点>"
```

### 多技术栈时

**并行模式**：多个技术栈同时启动代理并行执行，每个代理独立走 TDD 循环。
**顺序模式**：后端 → 前端，后端完成后输出数据存档供前端对齐。

## 第四步：验证循环

> 全部开发完成后，按顺序执行验证。通过时一行汇总，失败时修复后重跑。

### 4-A. 编译/构建验证

从 CLAUDE.md 获取构建命令并执行。失败时修复 1 轮，仍失败则等待用户。

### 4-B. Lint/Format 检查

仅对本次新建/修改的文件执行：
- 从 CLAUDE.md 或项目配置探测 lint/format 工具
- 自动修复格式问题
- error 级必须修复，warning 级列出

### 4-C. 测试验证

运行全部测试，确认：
- 本次新增的测试全部通过
- 已有测试未被破坏
- 覆盖率满足 `rules/{lang}/testing.md` 的要求

### 4-D. 安全验证

对照 `rules/common/security.md` + `rules/{lang}/security.md`：

**后端**：
- [ ] 无硬编码密钥
- [ ] 用户输入有参数校验
- [ ] 数据库查询参数化
- [ ] 错误响应不暴露内部信息
- [ ] 写操作有认证/授权
- [ ] 敏感操作有日志（不含密码）

**前端**：
- [ ] 无硬编码密钥
- [ ] 用户输入渲染前已转义
- [ ] API 请求携带认证
- [ ] 敏感数据不存 localStorage
- [ ] 表单有 CSRF 防护

### 4-E. 交叉验证（多技术栈时）

- 后端响应字段 vs 前端类型字段
- 后端接口路径 vs 前端 API 路径
- HTTP 方法一致性
- 请求参数名一致性
- 认证接口 vs 前端认证头

不一致项列出。

## 第五步：规则合规审查

> **写完代码后，回头审查是否按规则执行。**

### 5-A. 对照已加载规则逐项审查

遍历第一步加载的所有规则文件，逐项检查本次代码是否合规：

| 审查维度 | 对照规则 |
|---------|---------|
| 编码风格 | `rules/{lang}/coding-style.md` |
| 安全规范 | `rules/{lang}/security.md` |
| 设计模式 | `rules/{lang}/patterns.md` |
| 测试规范 | `rules/{lang}/testing.md` |
| 通用规范 | `rules/common/` 下所有规则 |
| 公共规范 | `common.md`（先读后写、改动最小化等） |
| 项目规范 | `{standards}/` 下的规范（若存在） |

### 5-B. 输出合规报告

```
【规则合规审查】
✅ coding-style: 命名规范一致、文件大小合理
✅ security: 无硬编码密钥、输入已校验
✅ patterns: 遵循 Repository/Service 分层
✅ testing: 覆盖率 85%、边界测试已覆盖
⚠ common/performance: 第 3 个接口未做分页（建议优化）
```

### 5-C. 输出最终报告

```
### 开发完成 ✓

【TDD 统计】
- RED: N 个测试先写后运行
- GREEN: N 个测试通过
- Git 检查点: N 个提交

【验证循环】
- 编译：✅
- Lint：✅（自动修复 N 处）
- 测试：✅（N 个通过，覆盖率 X%）
- 安全：✅（后端 N 项 / 前端 M 项）
- 交叉验证：✅/⚠

【规则合规】
- 已审查 N 个规则文件
- ✅ 通过 / ⚠ 建议

【文件清单】
▌ 按技术栈分组
- `路径` — 说明

【数据存档 — 供前端对齐】（后端完成时）
▌ 接口清单 + 响应字段 + 业务规则

下一步：/dev/review-implementation {name}
```

---

## 规则

- **第一步不可跳过**
- **TDD 是强制的**——没有失败的测试不写实现代码
- **不硬编码支持的语言**——有规则就遵循
- 规则冲突按优先级：项目 > 个人 > 插件默认
- 验证循环全部通过才能输出最终报告
- 规则合规审查是最后一道防线——写完代码后回头检查是否按规则执行
- Git 检查点在每个 TDD 循环后提交
