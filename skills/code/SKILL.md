# 写代码 — 统一代码编写

统一代码编写入口。动态发现并加载规则，根据项目实际技术栈执行开发。

> **本 skill 包含以下子文件**：
> - `common.md` — 公共开发规范（始终加载）
> - `backend.md` — 通用后端实现模板（所有后端语言通用）
> - `frontend.md` — 前端实现模板

## 使用方式

```
/code <功能名称或描述>
```

支持两种模式：
- **完整模式**：有设计文档时，按设计文档逐章实现
- **轻量模式**：无设计文档时，根据用户描述 + CLAUDE.md + 现有代码直接开发

---

## 执行流程

```
第零步：加载路径配置
    ↓
第一步：发现并加载规则 → 输出规则报告
    ↓
第二步：加载公共规范（common.md）
    ↓
第三步：读取材料 → 判断完整/轻量模式
    ↓
第四步：检测技术栈 → 明确时自动继续，有歧义时暂停确认
    ↓
第五步：探索现有代码模式
    ↓
第六步：执行开发（后端 → backend.md，前端 → frontend.md）
    ↓
第七步：合并与交叉验证（多技术栈时）
```

---

## 第零步：加载路径配置

> **最先执行，不可跳过。**

从用户的 `cc-skills.json` 读取路径变量。详见 `config/README.md`。

---

## 第一步：发现并加载规则

> **不可跳过。规则决定了支持哪些技术栈和遵循哪些规范。**
> **完成后立即输出规则加载报告。**

### 1-A. 按优先级扫描规则目录

| 优先级 | 位置 | 说明 |
|--------|------|------|
| 1（最高） | `{standards}` | 项目规范 |
| 2 | `{rules}` | 项目规则 |
| 3 | `{personal_rules}` | 个人规则 |
| 4（最低） | 本插件 `rules/` 目录 | 插件自带规则（兜底） |

### 1-B. 动态发现技术栈规则

扫描上述目录中存在的**子目录**（如 `java/`、`python/`、`golang/`、`typescript/`、`web/` 等），记录可用规则集合。

**不硬编码支持哪些语言**——有规则目录就支持，没有就不涉及。

### 1-C. 加载通用规则

始终加载 `rules/common/`。

### 1-D. 输出规则加载报告

```
【规则加载报告】
扫描目录：
- {standards}: 存在 / 不存在
- {rules}: 存在 / 不存在
- {personal_rules}: 存在 / 不存在
- 插件 rules/: 已加载

发现的语言规则：java, python, typescript, web, ...
加载的通用规则：common/ 下 N 个文件
冲突覆盖：{rules}/java/coding-style.md 覆盖了插件默认版本（若有）
```

## 第二步：加载公共规范

加载 `common.md`（4 规则 + 自检清单 + 错误预防）。

---

## 第三步：读取材料

### 3-A. 读 CLAUDE.md

读取项目 `CLAUDE.md`，了解技术栈、目录结构、构建命令。
读取 `{standards}` 下的代码规范。

### 3-B. 判断模式

检查 `{features}/{name}/dev-design/` 是否存在：

- **存在** → **完整模式**：读取设计文档全部内容，按章节提取接口、数据模型、UI 设计等
- **不存在** → **轻量模式**：使用用户在命令中的描述作为需求，结合 CLAUDE.md 和现有代码直接开发

### 3-C. 读需求文档（补充，完整模式下）

若 `{features}/{name}/requirement/` 存在，补充边界条件、必填规则、权限。

---

## 第四步：检测技术栈

结合第一步发现的规则和第三步读取的材料：

1. 从 CLAUDE.md 识别项目包含的技术栈
2. 从设计文档（完整模式）或用户描述（轻量模式）识别本次涉及的技术栈
3. 与第一步的可用规则取交集

**明确时自动继续**（只有一种技术栈或用户已指定）。
**有歧义时暂停确认**（检测到多种可能）。

---

## 第五步：探索现有代码模式

对每种检测到的技术栈：
- 读取该语言规则中的编码规范
- 在项目中找到至少 2 个同类现有文件作为模式参考

---

## 第六步：执行开发

根据检测到的技术栈路由：

| 类型 | 子文件 | 适用范围 |
|------|--------|---------|
| 后端 | `backend.md` | 所有后端语言（Java、Python、Go、Rust、Kotlin、C# 等） |
| 前端 | `frontend.md` | 前端开发 |

`backend.md` 提供语言无关的分层实现步骤（Model → DTO → Service → Controller），语言特定细节从 `rules/{lang}/` 获取。

### 多技术栈时

**并行模式**：多个技术栈同时启动代理并行执行。每个代理接收：
- CLAUDE.md 内容
- 设计文档（完整模式）或用户描述（轻量模式）
- 对应语言的规则文件
- 现有代码模式参考文件

**顺序模式**：按顺序执行后端 → 前端，后端完成后输出数据存档供前端对齐。

---

## 第七步：合并与交叉验证

多技术栈时执行：

1. **合并文件清单**：收集所有技术栈的新建/修改文件
2. **交叉验证**：
   - 后端响应结构字段名 vs 前端接口类型字段名 → 不一致则列出
   - 后端接口路径 vs 前端 API service 路径 → 不一致则列出
3. 不一致项列出，等待用户确认

```
下一步：/dev/review-implementation {name}
```

## 规则

- **第零步和第一步不可跳过**
- **不硬编码支持的语言**——有规则就遵循，没规则就不涉及
- 规则冲突按优先级：项目 > 个人 > 插件默认
- 规则加载报告在第一步完成后立即输出
- 所有路径从配置获取，不硬编码
- 公共规范统一执行一次
- 代码风格与现有代码保持一致
- 交叉验证是本 skill 的职责，子文件不做跨栈检查
