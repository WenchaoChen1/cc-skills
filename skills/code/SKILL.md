# 写代码 — 统一代码编写

统一代码编写入口。动态发现并加载规则，根据项目实际技术栈执行开发。

> **本 skill 包含以下子文件**：
> - `common.md` — 公共开发规范（始终加载）
> - `java.md` — Java 后端实现流程（有对应规则时加载）
> - `python.md` — Python 后端实现流程（有对应规则时加载）
> - `frontend.md` — 前端实现流程（有对应规则时加载）
> - `run.md` — 调度与合并逻辑

## 使用方式

```
/code <功能名称> [选项]
```

选项不限于特定语言，根据项目实际规则决定。

---

## 执行流程

```
第零步：加载路径配置
    ↓
第一步：发现并加载规则
    ↓
第二步：加载公共规范（common.md）
    ↓
第三步：读取材料（CLAUDE.md + 设计文档 + 需求文档）
    ↓
第四步：根据规则和项目检测技术栈
    ↓
第五步：探索现有代码模式
    ↓
第六步：执行开发（根据发现的规则路由）
    ↓
第七步：合并与交叉验证（run.md）
```

---

## 第零步：加载路径配置

> **最先执行，不可跳过。**
> 详见 `config/README.md`。

从用户的 `cc-skills.json` 读取路径变量。所有路径基于 `<project>` 或 `~` 拼接：`{基础路径}/{根变量}/{可配变量}/{name}/固定后缀`，每层可为空。

---

## 第一步：发现并加载规则

> **不可跳过。规则决定了支持哪些技术栈和遵循哪些规范。**

### 1-A. 按优先级扫描规则目录

按以下顺序扫描，**同名规则高优先级覆盖低优先级**：

| 优先级 | 位置 | 说明 |
|--------|------|------|
| 1（最高） | `{standards}` | 项目规范 |
| 2 | `{rules}` | 项目规则 |
| 3 | `{personal_rules}` | 个人规则 |
| 4（最低） | 本插件 `rules/` 目录 | 插件自带规则（兜底） |

### 1-B. 动态发现语言/技术栈规则

扫描上述目录中存在的**语言子目录**（如 `java/`、`python/`、`typescript/`、`golang/`、`rust/` 等），记录当前可用的技术栈规则集合。

**不硬编码支持哪些语言**——有规则目录就支持，没有就不涉及。

### 1-C. 加载通用规则

始终加载 `rules/common/`（或对应优先级目录中的通用规则）。

### 1-D. 规则冲突解决

同一规则文件（如 `coding-style.md`）在多个优先级目录都存在时：

```
项目规则 > 个人规则 > 插件默认规则
```

---

## 第二步：加载公共规范

> 加载本目录下 `common.md`，以下规则贯穿整个开发过程。

- **规则 A — 先读后写**：任何文件动笔之前，必须先读至少 2 个同类现有文件
- **规则 B — 改动最小化**：不修改无关文件，不做"顺便优化"
- **规则 C — 不确定时停下来问**：设计文档未覆盖的问题，暂停询问
- **规则 D — 输出可追溯**：每阶段输出文件清单和自检结果

详细规范见 `common.md`。

---

## 第三步：读取材料

### 3-A. 读 CLAUDE.md + 代码规范

读取项目 `CLAUDE.md`，了解技术栈、目录结构、构建命令。
读取 `{standards}` 下的代码规范。

> 后续所有路径、包名、框架版本以 CLAUDE.md 和实际代码为准，**不使用预设值**。

### 3-B. 读设计文档

读取 `{features}/{name}/dev-design/` 目录下所有文件。

### 3-C. 读需求文档（补充）

若 `{features}/{name}/requirement/` 存在，补充边界条件、必填规则、权限。

---

## 第四步：检测技术栈

结合第一步发现的规则和第三步读取的材料，判断本次开发涉及哪些技术栈：

1. **从 CLAUDE.md 识别**项目包含的技术栈
2. **从设计文档识别**本次功能涉及的技术栈
3. **与第一步的可用规则取交集**——只有存在对应规则的技术栈才纳入

输出检测结果，等待用户确认。

---

## 第五步：探索现有代码模式

按检测到的技术栈，读取对应参考文件。

对于每种技术栈：
- 读取该语言规则中的编码规范（`coding-style.md`）
- 在项目中找到 2 个同类现有文件作为模式参考

---

## 第六步：执行开发

根据检测到的技术栈，路由到对应子文件执行：

- 存在 Java 规则且涉及 Java → 执行 `java.md`
- 存在 Python 规则且涉及 Python → 执行 `python.md`
- 存在前端规则且涉及前端 → 执行 `frontend.md`
- 其他语言（如 Go、Rust 等）→ 根据对应规则和 `common.md` 的通用流程执行

### 并行模式
涉及多个技术栈时，可并行执行。

### 顺序模式
按顺序执行，每个技术栈完成后等待确认。

---

## 第七步：合并与交叉验证

> 使用 `run.md` 中的合并逻辑。

1. 合并所有技术栈的文件清单
2. 交叉验证（如有多个技术栈）：
   - 后端字段名 vs 前端字段名
   - 后端接口路径 vs 前端 API 路径
3. 不一致项列出，等待用户确认

## 规则

- **第零步加载配置、第一步发现规则均不可跳过**
- **不硬编码支持的语言**——有规则就遵循，没规则就不涉及
- 规则冲突按优先级：项目 > 个人 > 插件默认
- 所有路径从配置获取，不硬编码
- 公共规范统一执行一次，不在每个技术栈中重复
- 代码风格与现有代码保持一致
- 交叉验证是本 skill 的职责，子文件不做跨栈检查
