# 公共开发规范

> 所有开发子模块（backend / frontend）必须遵守的规则。
> 编码质量规范（命名、格式化、安全等）见 `rules/common/` 和对应语言 `rules/{lang}/` 目录。

---

## 规则

### 规则 A：先读后写

- 任何文件动笔之前，**必须先读至少 2 个同类现有文件**作为模式参考
- 禁止凭印象猜测现有代码的包名、类名、字段命名风格
- 先阅读项目的 `CLAUDE.md`，了解技术栈、目录结构、模块组织方式

### 规则 B：改动最小化

- 不修改无关文件；不重构未涉及的代码
- 新增文件优先放进现有模块目录，不新建顶层目录
- 不添加未使用的依赖、不引入不必要的新包

### 规则 C：遇到不确定的设计点，停下来问

- 如果设计文档与现有代码模式有明显冲突，列出冲突点，询问用户如何取舍
- 不要自行决定推翻现有架构
- 如果某个章节设计文档未覆盖且无法从现有代码推断，停下来问

### 规则 D：输出可追溯

- 每个阶段完成后必须输出文件清单（新建/修改）
- 自检结果逐项列出
- 有疑问或取舍的地方标注 `⚠️`

---

## 通用自检清单

| 检查项 | 说明 |
|--------|------|
| 包名/路径正确 | import 无遗漏，模块路径与现有代码一致 |
| 命名风格一致 | 参照同模块现有文件的命名惯例 |
| 无编译/语法错误 | 模块路径、类型定义、括号匹配 |
| 无未使用的 import | 清理多余引入 |
| 改动范围受控 | 未修改无关文件 |

---

## 常见错误预防

| 错误类型 | 预防措施 |
|---------|---------|
| 包名/模块路径猜错 | 开始前读同模块现有文件确认路径 |
| 前后端字段名不一致 | 逐字段核对后端 DTO/Schema 和前端 interface |
| 架构冲突自行决定 | 发现冲突时停下列出，询问用户后再动手 |
| 映射层漏转字段 | 使用项目现有的映射工具和配置风格 |
