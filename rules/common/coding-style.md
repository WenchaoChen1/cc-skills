# 编码风格

## 不可变性（关键）

始终创建新对象，绝不修改现有对象：

```
// Pseudocode
WRONG:  modify(original, field, value) → changes original in-place
CORRECT: update(original, field, value) → returns new copy with change
```

原因：不可变数据可防止隐藏的副作用，使调试更加容易，并支持安全的并发操作。

## 核心原则

### KISS（保持简单）

- 优先选择真正有效的最简方案
- 避免过早优化
- 清晰优先于巧妙

### DRY（不要重复自己）

- 将重复逻辑提取到共享函数或工具模块中
- 避免复制粘贴导致的实现偏差
- 当重复确实存在（而非推测）时再引入抽象

### YAGNI（你不会需要它）

- 不要在需要之前构建功能或抽象
- 避免投机性的过度泛化
- 先从简单开始，当真正有压力时再重构

## 文件组织

多个小文件 > 少数大文件：
- 高内聚、低耦合
- 通常 200-400 行，最大 800 行
- 从大模块中提取工具函数
- 按功能/领域组织，而非按类型组织

## 错误处理

始终全面处理错误：
- 在每个层级显式处理错误
- 在面向用户的代码中提供友好的错误消息
- 在服务器端记录详细的错误上下文
- 绝不静默吞掉错误

## 输入校验

始终在系统边界进行校验：
- 处理前校验所有用户输入
- 尽可能使用基于 schema 的校验
- 快速失败并给出清晰的错误消息
- 绝不信任外部数据（API 响应、用户输入、文件内容）

## 命名规范

- 变量和函数：使用描述性名称的 `camelCase`
- 布尔值：优先使用 `is`、`has`、`should` 或 `can` 前缀
- 接口、类型和组件：`PascalCase`
- 常量：`UPPER_SNAKE_CASE`
- 自定义 hooks：使用 `use` 前缀的 `camelCase`

## 需要避免的代码异味

### 深层嵌套

当逻辑开始堆叠时，优先使用提前返回而非嵌套条件。

### 魔法数字

为有意义的阈值、延迟和限制使用命名常量。

### 过长函数

将大函数拆分为职责清晰的小函数。

## 代码质量检查清单

在标记工作完成之前：
- [ ] 代码可读性好且命名规范
- [ ] 函数短小（<50 行）
- [ ] 文件职责集中（<800 行）
- [ ] 无深层嵌套（>4 层）
- [ ] 错误处理得当
- [ ] 无硬编码值（使用常量或配置）
- [ ] 无可变操作（使用不可变模式）
